# 当前项目架构说明

## 目的

本文档用于说明当前项目的整体架构，重点描述多轮 agent 运行时、semantic-state 机制、工具系统、日志系统和 GDPVal 评测外壳之间的关系。内容聚焦模块职责、输入输出、标签体系和处理流程，不展示代码，不按源码位置展开。

## 一、整体架构概览

当前项目不是一个单纯的“模型 + 工具调用”系统，而是一个分层的 agent 运行框架，并在外层接了 GDPVal 评测壳。

整体可以分成以下八层：

1. 客户端层
2. 共享模型层
3. Agent 编排层
4. 规划层
5. Semantic-State 层
6. 工具层
7. 日志与可观测性层
8. 评测层

这八层共同构成了当前系统的实际运行架构。

## 二、一次任务的完整执行链路

一条标准任务在当前系统中的流转顺序如下：

1. 用户任务进入 agent 运行时。
2. 规划层先把任务转成结构化计划。
3. Agent 建立 session，并初始化工具和执行环境。
4. 系统提示词、任务提示词和输入文件上下文被组装。
5. Semantic-State 层把阶段状态、当前状态、激活规则注入给下一轮模型。
6. 模型输出 assistant 消息，可能附带工具调用。
7. 工具层执行相应工具，返回工具结果。
8. Semantic-State 层解析 assistant 显式输出的状态更新。
9. Extractor 从 assistant 输出和工具结果中补提取隐含状态。
10. Auditor 检查当前回合是否违反了激活规则。
11. 更新后的状态再次注入下一轮。
12. 任务结束时，finish 工具验证输出文件是否存在，并正式结束任务。
13. 日志层和评测层记录轨迹、统计信息和最终结果。

这说明 semantic-state 并不是独立服务，而是直接嵌在 agent 主循环中，每轮都参与运行。

## 三、各层模块说明

### 1. 客户端层

客户端层负责把不同模型提供方统一包装成同一种生成接口。

它的职责包括：

- 接收标准化消息
- 向模型后端发起生成请求
- 返回统一格式的 assistant 响应
- 记录 token 使用量和请求时间

这一层处理的主要消息角色标签是：

- `system`
- `user`
- `assistant`
- `tool`

这些标签的意义是消息协议角色，不是业务语义。

当前客户端层不仅服务主 agent，也服务 planner、extractor 和 auditor，因此它是整个系统的统一推理入口。

### 2. 共享模型层

共享模型层定义了整个系统通用的数据结构，相当于协议层。

它主要负责统一以下对象：

- 对话消息
- 工具调用请求
- 工具执行结果
- token 使用统计
- 子 agent 执行元数据
- Tool 抽象
- ToolProvider 抽象

这一层的作用是让上层所有模块都通过同一种数据协议互相通信。

### 3. Agent 编排层

这是整个系统的核心执行层。

它负责：

- 创建和管理 session
- 初始化和销毁工具
- 挂载输入文件
- 组装完整提示词
- 驱动多轮对话循环
- 解析工具调用
- 执行工具
- 处理 semantic-state 注入
- 触发上下文压缩
- 在完成时调用 finish

这一层管理两类核心标签。

第一类是运行控制标签：

- 当前 turn
- 剩余 turn
- 是否接近 turn 上限
- 是否已 finish
- 是否进入上下文压缩

第二类是语义状态注入标签：

- `semantic_state_notices`
- `phase_status`
- `current_state`
- `active_rules`

这一层可以理解成整个项目的任务操作系统。

### 4. 规划层

规划层在正式执行前先运行一次，把自然语言任务转换成结构化计划。

它处理的主要标签包括：

- `task_understanding`
- `deliverables`
- `phases`
- `total_turn_budget`
- `key_rules`
- `risk_flags`

其中，deliverable 主要包含：

- `name`
- `format`
- `constraints`

phase 主要包含：

- `name`
- `goal`
- `turn_budget`
- `variables_to_extract`

规划层的处理方式是：

- 先请求模型返回结构化 JSON
- 如果 JSON 解析或校验失败，则走 fallback plan

fallback plan 至少会生成两个阶段：

- 任务分析阶段
- 执行阶段

因此，规划层是一个“模型生成 + 兜底模板”的混合设计。

## 四、Semantic-State 层

Semantic-State 是当前项目里最重要的实验性增强模块。它试图解决多轮 agent 在长轨迹中丢失事实、忘记约束、偏离任务目标的问题。

这一层由以下几个部分组成：

- 语义数据模型
- 状态管理器
- 状态更新解析器
- stale 传播机制
- 回合上下文序列化器
- Extractor
- Auditor
- 纠偏 notice 机制

### 1. Semantic-State 里的主要语义标签

当前系统里最重要的语义变量类别有四种：

- `data_fact`
- `task_constraint`
- `intermediate_result`
- `env_state`

这四类是 semantic-state 真正的业务语义标签。

#### `data_fact`

表示稳定事实，例如：

- 文档中的价格
- 数量
- 日期
- lead time
- 合同金额
- 明确写出的客观结论

处理方式：

- 可以由 agent 显式声明
- 可以由 extractor 从工具结果中补提取
- 如果后续被覆盖，会记录覆盖历史
- 如果其他变量依赖它，这些派生变量可能被标记为 stale

#### `task_constraint`

表示任务约束，例如：

- 交付物格式要求
- 输出语言要求
- 优先级要求
- 先做什么后做什么的要求

处理方式：

- 一般来自 planner 或 rule enforcement
- 主要作用是约束后续行为
- 更偏执行控制，不直接用于最终答案内容

#### `intermediate_result`

表示中间结果，例如：

- 计算结果
- 比较结果
- 阶段性结论
- 尚未写入最终交付物的分析结果

处理方式：

- 理论上应由工具输出或分析步骤写入
- 用于推进 phase completion 和后续任务执行
- 当前系统对这类变量的落账还不够稳定

#### `env_state`

表示环境事实，例如：

- 输入文件是否已在当前工作目录可见
- 某类路径是否可用
- 工具能力是否受限
- 当前执行环境有哪些约束
- 输出目录、执行环境工作目录和路径使用策略
- 浏览器、Web、代码执行、图像查看和 finish 工具是否可用
- 最近一次工具失败、缺失依赖、生成文件列表和 finish 路径校验结果

处理方式：

- 由 runtime 优先自动写入，而不是主要依赖模型声明
- session 初始化后自动记录 `env.output_dir`、`env.exec_environment`、`env.exec_cwd`、`env.uploaded_files`、`env.tool_availability` 和 `env.browser`
- 每轮工具调用后自动扫描执行环境文件，更新 `env.workspace_files` 和 `env.generated_files`
- 工具失败时自动记录 `env.last_tool_error`，并根据错误内容识别 `missing_dependency`、路径错误、权限错误、超时和网络/Web 错误
- finish 工具调用后自动记录 `env.finish_status`、`env.finish.reported_paths`、`env.finish.validated_paths` 和 `env.finish.missing_paths`
- runtime 写入的 env_state 使用更高可信优先级，agent 或 extractor 不能随意覆盖 runtime、tool_runtime 或 finish_tool 已验证的环境事实
- 对缺依赖、工具失败和 finish 失败会生成 `auto_correction` notice，下一轮模型必须先修复具体环境问题

### 2. 规则标签

Semantic-State 层还维护规则对象。

每条规则包含：

- `rule_id`
- `category`
- `text`
- `resolution_strategy`

规则类别用于区分规则来源或性质，比如：

- 显式要求
- 隐含约束
- 交付物要求

常见策略有：

- `auto_patch`
- `rollback`

在审计阶段还会出现：

- `follow_rule`
- `circuit_breaker`

### 3. 交付物标签

每个交付物对象包含：

- `name`
- `format`
- `constraints`

它们定义了系统预期最终要产出的内容。

### 4. 阶段标签

每个阶段对象包含：

- `name`
- `goal`
- `turn_budget`
- `variables_to_extract`

这是日志中 `phase_status` 能显示阶段进度的基础。

## 五、Semantic-State 管理器

状态管理器是 Semantic-State 的执行核心。

它维护以下核心内部对象：

- 激活规则
- 阶段调度表
- planned variables
- discovered variables
- 当前状态字典
- turn 快照
- pending notices
- pending auto corrections
- 规则重试计数
- unresolvable rules
- phase warning 发射记录

这些对象共同支持六个子功能模块。

### 1. 阶段跟踪子模块

这个子模块负责根据 turn 计算当前所处阶段，并生成 `phase_status`。

`phase_status` 主要包含：

- 当前 phase 名称
- 当前 phase 已使用 turn 数
- 当前 phase 剩余 turn 数
- 每个 planned variable 的进度状态

处理方式：

- planner 输出的 `variables_to_extract` 仍保持字符串格式，但 runtime 内部会归一化成 planned variable spec
- 每个 planned variable 会维护 `pending`、`candidate`、`satisfied`、`blocked`、`stale` 等状态，而不是简单布尔值
- 同名变量、别名变量或 extractor 显式绑定的变量，都可以成为 planned variable 的候选状态变量
- 单纯出现在工具 stdout 中，只会作为 evidence 记录，不会自动算完成
- 对路径类 planned variable，例如 `final_deliverable_path`，需要 runtime 文件扫描或 finish 校验等硬环境事实参与验证
- 如果阶段结束仍有缺项，会触发带细节的 phase warning，说明是缺状态、候选未验证，还是状态已失效

### 2. 当前状态子模块

这个子模块负责保存当前所有语义变量，并在下一轮注入给模型。

它生成的主要标签是：

- `current_state`
- `STALE ...`
- `written_turn`
- `confidence`
- `overwritten`

处理方式：

- 非 stale 状态表示当前可直接复用的上下文
- stale 状态表示它曾经成立，但现在不能无条件继续使用
- 如果一个变量被多次覆盖，会保留覆盖计数

### 3. 激活规则序列化子模块

这个子模块把当前有效规则整理成 `active_rules` 注入给模型。

每条规则序列化后会带：

- 规则标识
- 规则类别
- 解决策略
- 规则文本

处理方式：

- 规则通过 prompt 约束模型
- 它们更接近软约束，而不是硬拦截

### 4. 状态更新解析子模块

这个子模块负责解析 assistant 输出中的显式状态更新。

它识别的标签包括：

- `state_update`
- `variable`
- `name`
- `value`
- `category`
- `source`
- `update_mode`
- `reason`
- `derived_from`
- `evidence_refs`
- `matched_planned_variable_id`

处理方式：

- 只有结构合法的状态更新才会被接收
- 接收后写入状态字典
- 如果变量名或 `matched_planned_variable_id` 命中了 planned variable，会先绑定为候选变量
- 是否最终标记 `satisfied` 由 planned variable 的 completion policy 和 verifier 决定

### 5. Stale 传播子模块

这个子模块负责处理依赖失效。

它依赖的关键标签是：

- `derived_from`
- `stale`

处理方式：

- 如果某个基础变量被覆盖，依赖它的派生变量可以被标 stale
- 这样系统能保留“哪些结论已失效”的信息

这是目前 semantic-state 设计里相对更成熟的一部分，因为它真正表达了变量间依赖关系。

### 6. 纠偏 notice 子模块

这个子模块负责生成纠偏类提示。

它会注入的标签包括：

- `semantic_state_notices`
- `auto_correction`
- `state_rollback`
- `phase_warning`
- `circuit_breaker`
- `warn`

这些标签分别表示：

#### `auto_correction`

表示系统认为上一轮偏离了某条规则，下一轮应该修正。

#### `state_rollback`

表示系统希望下一轮按某个旧状态快照重新约束执行。

#### `phase_warning`

表示阶段结束或即将结束，但应提取变量仍未补齐。

#### `circuit_breaker`

表示某条规则反复修不回来，系统将其视为不可解。

#### `warn`

表示较轻量的提醒，比如覆盖已有事实。

这套机制让 semantic-state 从被动记忆层变成了主动监督层。

## 六、Extractor 与 Auditor

当前系统比最初的“状态注入”方案更复杂，已经发展出两个额外的模型辅助回路。

### 1. Extractor

Extractor 是主回合结束后额外触发的一次模型调用。

它的职责是从以下来源补提取语义变量：

- assistant 的自然语言回复
- 工具执行结果

它提取的字段仍然是统一结构：

- `name`
- `value`
- `category`
- `source`
- `update_mode`
- `reason`
- `derived_from`

它的架构作用是：

- 弥补 agent 没有显式输出 `state_update` 的情况
- 把轨迹里已经存在但没落账的信息补写进 semantic-state

当前限制：

- 它容易把“隐含推断”写成“稳定事实”

### 2. Auditor

Auditor 也是每轮后触发��一次模型调用。

它负责检查当前回合是否违反了 active rules。

它输出的主要字段包括：

- `rule_id`
- `severity`
- `message`
- `resolution_strategy`
- `rollback_turn`
- `suggested_fix`

它的架构作用是：

- 发现规则违背
- 为运行时提供后续纠偏动作

可能的处理结果包括：

- 无动作
- 自动修补
- 回滚
- 进入 circuit breaker

因此，当前系统已经不是简单的对话循环，而是带监督回路的 agent 循环。

## 七、工具层架构

工具层分成两种形态：

- 静态工具
- 带生命周期的 ToolProvider

Provider 型工具更重要，因为很多工具需要持有执行环境、HTTP 连接、浏览器会话或服务端连接。

### 1. 代码执行模块

代码执行模块提供命令运行能力，支持多种后端风格，包括本地执行和更隔离的执行方式。

它常见的输出标签包括：

- 标准输出
- 标准错误
- 退出状态
- 错误类型
- 工具元数据

这一模块的作用是：

- 处理本地分析
- 生成交付物
- 做格式转换
- 执行临时脚本

它是 reasoning 到 artifact creation 的主桥梁。

### 2. Web 模块

Web 模块提供网页抓取和搜索能力。

它主要处理的标签包括：

- URL
- 页面正文
- 搜索结果
- 成功与���败状态

它的作用是：

- 获取在线资料
- 为 agent 提供远程信息补充

### 3. Browser 模块

Browser 模块提供更重型的浏览器交互能力，适用于简单抓取无法满足的页面交互场景。

### 4. 图像查看模块

该模块允许 agent 读取图像输入内容。

### 5. 用户输入模块

该模块允许 agent 在必要时向用户请求额外信息。

### 6. Finish 模块

Finish 模块是任务正式完成的出口。

它处理的核心标签有：

- `reason`
- `paths`
- `success`

处理方式：

- 不是模型说“完成了”就算完成
- 系统会检查上报的输出文件是否真实存在
- 文件不存在时，finish 失败

因此，finish 是一个带验证的完成协议，而不是一句自然语言声明。

## 八、日志与可观测性层

日志层不是简单打印，而是一个结构化运行观察器。

它记录：

- task 开始
- 用户消息
- assistant 消息
- 工具调用
- 工具结果
- warning
- finish 原因
- token 使用量
- 工具耗时
- 模型速度统计
- 上下文压缩事件
- semantic-state 事件

它的架构作用有三层：

- 展示层
- 诊断层
- 元数据汇总层

对于当前 autoresearch 方向来说，这一层尤其重要，因为 semantic-state 的效果很大程度上要靠日志和元数据来观察。

## 九、评测层

GDPVal 评测层是整个系统的外部运行壳。

它负责：

- 加载任务
- 过滤任务子集
- 下载参考文件
- 准备任务输出目录
- 创建 agent
- 挂载所需工具
- 执行任务
- 收集完成原因
- 记录成功或失败
- 触发评分和结果汇总

这一层本身不实现 semantic-state 逻辑，它只是调用已经增强过的 agent runtime。

## 十一、当前架构的真实形态

从实际运行方式看，当前项目最准确的描述是：

- 一个统一协议驱动的多轮 agent runtime
- 配套一个带生命周期管理的工具系统
- 前面接一个 planner
- 中间接一个 semantic-state 记忆与进度层
- 后面再叠一个 extractor
- 再叠一个 auditor
- 外层包一个评测壳

它的主要优点是：

- 分层比较清楚
- 协议对象统一
- 工具生命周期管理明确
- semantic-state 已经独立成模块，不是零散逻辑

它的主要弱点是：

- semantic-state 目前更像监督层，而不是强执行层
- env_state 已经开始承担 runtime 事实落账和路径校验职责，但仍需要更多任务类型的 verifier policy
- planned variable 已经不再只依赖精确命名；当前支持别名绑定、evidence 记录和路径类 runtime verifier，但复杂语义变量的 verifier 仍需要继续扩展
- extractor 和 auditor 让系统开始出现“第二隐藏控制回路”的复杂度问题

## 十二、总结

当前项目已经演化成一个多回路 agent 框架。主干仍然是 agent runtime，但 semantic-state 已经成为决定执行质量的重要增强层。它负责保存任务记忆、表达阶段进度、注入执行约束，并通过 extractor、auditor 和纠偏 notice 对每一轮进行监督。

因此，当前系统的真实定位不是“能调工具的 LLM”，而是“带规划、记忆、监督、工具和评测壳的多轮 agent 运行架构”。
